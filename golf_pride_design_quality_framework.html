The Problem
Where quality breaks down

Most teams don't lose quality because of bad intentions — they lose it because of three specific, fixable failure points in the design-to-production pipeline. Each one compounds the others.

01
Translation Loss
What leaves Figma and what gets built are subtly different — spacing, type scale, corner radius, interactive states. Nobody caught it because nobody was explicitly responsible for the comparison.
02
Definition Gaps
The design showed the happy path. Developers had to improvise edge cases, empty states, error states, and loading states. Those improvised decisions accumulate into a product that feels "off" without a specific mistake to point to.
03
Late Discovery
QA and UAT happen after the build is "done," so every fix requires rework — not correction. The cost and team friction at this stage is 5–10x what it would be if caught at 50% build.
⚡
Quality becomes "everyone's job" — which quietly becomes "no one's job." The frameworks that follow attack this directly with clear ownership at every stage, not shared ambiguity.
The Framework
Four phases, clear owners

The framework organizes the design-to-production workflow into four distinct phases — each with a primary owner and defined handoff points. This replaces a single vague "handoff" moment with a continuous quality thread.

Note: Design QA and Functional QA are deliberately separated — they require different references and different skill sets. Collapsing them is where things fall through the cracks.

The Rollout
Crawl → Walk → Run

The full framework introduced at once creates adoption friction and usually collapses. Instead, we phase it across three maturity stages — each building on the last, each delivering immediate value, and each calibrated to what the team can realistically absorb.

Crawl
Foundations
Establish shared language, close the most obvious handoff gaps — zero new tooling required.
Weeks 1–6
Goal: Create a baseline of shared expectations without disrupting the existing workflow. Every tactic here can be adopted in the next sprint.
Tactics
  • Define "design complete" — create a 5-item completeness checklist as a required sub-task on every design ticket before it enters the dev queue
  • Run the first handoff alignment session for one feature this sprint — 30 minutes, designer walks developer through intent and behavior
  • Designer performs one 50% build check-in per sprint, calendared, not ad-hoc
  • Separate Design QA and Functional QA as two distinct sub-tasks in your PM tool so neither gets skipped in the rush to close
  • Introduce the quality matrix — a shared doc showing each feature's QA status across functional, design, and UAT streams
  • Begin logging handoff decisions: any time a developer makes a UI call the design didn't specify, it's logged in the ticket comment
Dependencies & Asks
  • PMCommit to enforcing the completeness gate — no ticket moves to dev without the checklist passing
  • DesignerBudget ~1.5 hrs/sprint for the alignment session and 50% check-in
  • DevAgree to log improvised UI decisions in ticket comments rather than silently deciding
  • QASeparate design QA reviews from functional QA in your workflow, even if the same person runs both
  • ToolExisting PM tool (Linear / Jira / Notion) — no new tools needed in this phase
Walk
Process Solidification
Make crawl-phase habits reliable, add lightweight tooling, connect design tokens to code.
Months 2–4
Goal: The crawl habits are now assumed, not negotiated. This phase hardens them into workflow infrastructure and introduces the token pipeline that makes design QA faster and more objective.
Tactics
  • PM formally enforces the completeness gate — tickets without it are returned, not fast-tracked
  • The Design QA Checklist (Section 5) is introduced as the official design review standard for every release
  • Decision log moves from ticket comments to a structured Notion page or Figma annotation layer per feature
  • Tokens Studio → Style Dictionary pipeline established: design tokens in Figma sync to code variables, making QA token-name-based rather than eyeball-based
  • Introduce Storybook for component documentation — each new component has a story before design QA closes
  • Hold a team retrospective at month 3 — what's working, what's been skipped, what needs refinement
  • UAT structure formalized: PM prepares scenario-based testing scripts instead of open-ended "take a look" reviews
Dependencies & Asks
  • LeaderFigma organization plan or admin access for proper component library publishing and Tokens Studio integration
  • LeaderBudget approval for Tokens Studio (free tier available, paid at $0–$15/mo) if not already licensed
  • DevSet up Style Dictionary pipeline in the codebase — approximately 2–4 hrs one-time investment
  • DesignerMigrate Figma styles to variables and Tokens Studio format — approximately 4–8 hrs depending on existing system maturity
  • QAFamiliarize with the design QA checklist and apply it in reviews — training session recommended
  • PMDesign and run the first structured UAT scenario set for Golf Pride stakeholders
Run
Systemic Quality
Quality is built into the system — automated, documented, and self-reinforcing.
Month 5+
Goal: Quality requires no heroics. The pipeline catches regressions automatically, new team members onboard to the process quickly, and the team can release with confidence.
Tactics
  • Figma Code Connect operational — developers reference live component code from design; design and code are the same source of truth
  • Visual regression testing via Chromatic (Storybook) or Percy runs on every PR — catches unintended visual changes automatically
  • Design QA checklist evolves: token-name verification replaces eyeball checks, reducing review time from ~40 min to ~15 min
  • Accessibility automated testing (axe-core or Deque) integrated into CI pipeline as a hard gate on PRs
  • Onboarding documentation for new team members covers the full quality process — no tribal knowledge dependencies
  • Quarterly "design health audit" — a structured review of design debt, token drift, and component divergence across the product
Dependencies & Asks
  • LeaderBudget for Chromatic ($0–$149/mo depending on snapshots) or Percy — essential for visual regression at this stage
  • LeaderGitHub CI/CD integration permissions for QA gates on pull requests
  • DevChromatic or Percy integration into CI pipeline — approximately 1 day setup
  • DevCode Connect configuration for all current Figma components — ongoing, ~1 hr per component
  • DesignerMaintain Figma component library as the active source of truth — not a one-time exercise but an ongoing commitment
  • PMDocument and maintain the quality process; update it after every retrospective
The Case for It
Why this investment pays off

The crawl phase alone — which requires no new tools or budget — typically reduces design rework in QA by 40–60% simply by making issues discoverable earlier. Here's the full picture of what changes.

⚙️
Fewer late-stage surprises
Issues caught at the 50% check-in are corrected, not reworked. The cost of a fix at this stage is typically 10–20% of what it costs after dev-complete — and it produces no stakeholder visibility or anxiety.
🎯
Higher first-pass release quality
When design QA runs against a specific checklist, defects are caught consistently — not by whoever happens to notice something looks wrong. Production releases start matching Figma spec more reliably.
🤝
Stronger client trust
Golf Pride stakeholders experience fewer "that's not what we signed off on" moments in UAT. Structured UAT scenarios replace open-ended review, reducing subjective scope creep and keeping feedback actionable.
🧠
Less tribal knowledge dependency
Decision logs and documented processes mean a new developer or QA person can get up to speed quickly. Quality doesn't live only in individual heads.
📐
Design system actually used
The token pipeline and Code Connect integration ensure the design system is a living reference — not a Figma file that slowly diverges from what's in production.
🔁
Sustainable pace
The crawl/walk/run structure means no single sprint absorbs all the process change. The team builds habits incrementally, which is the only way process changes actually stick.
Stakeholder Guide
What we need from each player

This framework only works with aligned expectations across roles. Here's what each player is being asked to commit to — and what they get in return.

Role
Product Manager
  • Enforce the completeness gate — return tickets that aren't spec'd
  • Calendar the handoff session and 50% check-in as recurring tasks
  • Own UAT facilitation and triage feedback into bugs vs. scope vs. preference
  • Maintain and share the quality matrix with the full team
  • Secure tool access and budget approvals for walk/run phases
Role
Designer
  • Deliver design-complete specs before tickets enter the dev queue
  • Define all interactive states, edge cases, and responsive behavior — not just the happy path
  • Show up for the handoff session and the 50% check-in (~1.5 hrs/sprint)
  • Own Design QA as a separate pass against the checklist (Section 5)
  • Migrate Figma styles to variables/tokens during walk phase
Role
Developer
  • Actively participate in handoff sessions — ask questions, surface ambiguity before building
  • Log improvised UI decisions in ticket comments rather than deciding silently
  • A ticket isn't "done" until design QA is passed — update the definition of done
  • Set up Style Dictionary pipeline and Storybook during the walk phase
  • Integrate visual regression and accessibility tooling in the run phase
Role
QA
  • Treat Functional QA and Design QA as separate passes with separate references
  • Functional QA runs against ticket acceptance criteria, not visual judgment
  • Support designer with QA environment and device access during Design QA
  • Participate in UAT scenario design so testing is structured from the start
  • Own accessibility testing as part of functional QA baseline
🔑
For leadership / tool approvals specifically: The crawl phase requires no new tools or budget. The walk phase requires Figma organization plan access (if not already in place) and potentially a Tokens Studio license. The run phase requires CI integration permissions and a budget for visual regression tooling (Chromatic or Percy). These are the three asks that need leadership alignment before the walk and run phases begin.
Section 5 — Design QA
The Design Testing Checklist

Use this during the Design QA pass — after functional QA, before UAT. The designer owns this review. Items are ordered to match how a reviewer naturally moves through a feature: preconditions → visual fidelity → behavior → resilience → environment → accessibility. Items are interactive — check them off as you go.

Before you start
Pre-flight Gate
Confirm all five conditions are true before opening the checklist. If any are false, return the build — design QA on an incomplete build produces noise, not signal.
The feature is dev-complete — not "mostly done." All screens and states are built.
Real or realistic content is loaded. No lorem ipsum, placeholder images, or "TBD" copy.
All assets are loading: fonts, icons, and images are rendering — no broken asset requests in the console.
The designer has access to the specific build URL or branch being reviewed.
The corresponding Figma file is open and at the correct version — not a stale or in-progress frame.
Token Adherence Principle — applies to every section below. All visual values — spacing, color, typography, elevation, border radius, and shadow — must reference named design tokens. Any hardcoded value that doesn't map to a token requires explicit designer sign-off and a decision log entry. This principle is stated once here and assumed throughout; individual sections will not repeat it.
0 of 58 items checked
📐 Spacing & Layout 5 items
Container widths and max-widths match Figma.High Verify the primary content container width, section max-widths, and any fixed-width elements against the Figma frame dimensions. Use browser dev tools to inspect computed widths — not a visual guess.
Alignment is exact — not just "close."High Verify flex/grid alignment against the Figma spec using dev tools or a pixel overlay. Pay extra attention to icon-to-label alignment, baseline alignment in multi-column rows, and inline element spacing — these are the most commonly eyeballed and most commonly wrong.
Visual hierarchy matches the intended reading order.Medium Confirm that size, weight, and spacing relationships between heading levels, body copy, and supporting text produce the same visual hierarchy as Figma. Check at 100% zoom — designers work at 100%, reviewers should too.
Component internal spacing is consistent across all instances. Verify that padding within repeating components (cards, list items, table rows) is uniform across every instance on the page — not just the first one. Dynamic content length can silently collapse spacing in later instances.
Stacking order and z-index layering match design intent. Modals, tooltips, dropdowns, sticky headers, and overlapping elements appear in the correct visual layer. Verify no unintended element clipping, and that overlays properly obscure background content.
🔤 Typography 5 items
Font loads correctly and doesn't cause visible reflow.High Throttle the network in dev tools and confirm that the web font loads without a jarring flash of unstyled text (FOUT) or layout shift as the font metric changes. Verify the fallback font stack is set — not just "sans-serif" — so the fallback is close in proportion to the web font.
Letter spacing, text-transform, and font variant settings are applied correctly. Uppercase labels, headline tracking, and small-caps text are commonly dropped in the build. Verify these against Figma individually — they don't transfer through copy-paste and are easy to miss.
Long text wraps correctly; short text doesn't create layout gaps.High Test each text element with both minimum content (1–2 words) and maximum realistic content (2–3× expected length). Verify overflow behavior — truncation with ellipsis, line clamping, or full wrap — matches the Figma spec and doesn't break the surrounding layout.
Line height doesn't clip descenders or create tight paragraph spacing. Check that tighter line heights on headlines don't clip letters like g, p, y. Verify paragraph spacing between body copy blocks matches Figma — especially in CMS-driven or rich text areas where spacing is often reset by the editor.
Primary content (heroes, headlines, pull quotes) wraps cleanly across breakpoints. Verify that hero text and section headers don't produce awkward one-word final lines (widows) or odd breaks at tablet and mobile sizes. These are rarely specced but always noticed.
🎨 Color & Contrast 4 items
Brand-critical colors verified on screen, not just in Figma.High Colors render differently in browser vs. Figma due to color profile differences. Do a side-by-side comparison of Golf Pride brand colors (greens, accents) between Figma and the browser at 100% zoom. Verify brand colors match the Golf Pride brand spec exactly — "close enough" ships as wrong.
Text contrast passes WCAG AA at minimum — in all states.Critical Body text / background combinations require 4.5:1; large text (18px+ regular, 14px+ bold) requires 3:1. Check every interactive state — hover, focus, disabled — not just default. Note: the automated axe audit in the Accessibility section also catches contrast failures; don't run both manually.
Theme switching (if applicable) applies completely — no orphaned hardcoded values. If the design supports dark/light mode, verify that every element responds correctly to theme change, including SVGs with inline fill colors, images with text overlays, and gradient backgrounds. Any hardcoded color value will not switch.
Status and state differentiation doesn't rely on color alone. Error, success, warning, and disabled states each use a secondary differentiator — icon, label, pattern, or texture — in addition to color. This is both an accessibility requirement and a UX quality bar.
🧩 Component Fidelity 5 items
All component variants are implemented — none silently omitted.Critical Every variant in Figma (size: sm/md/lg, style: primary/secondary/ghost, type: icon-left/icon-only) has a built counterpart. If a variant was deferred, it must be explicitly flagged in the ticket — not quietly dropped.
All interactive states verified individually.Critical For every interactive component: default, hover, focus, active/pressed, disabled, loading, and selected/checked — each triggered and compared to Figma. States not specced in Figma must be confirmed with the designer before design QA closes; developer-improvised states are a QA failure, not an open question.
Icon sizing, stroke weight, and optical centering match the spec. Icons are the correct size variant (16/20/24px), use the correct stroke weight for the context, and are optically — not just mathematically — centered within their container. A 24px icon in a 48px button that isn't optically centered is a common failure.
Net-new components (outside the design system) are flagged and reviewed. If a developer built a component that doesn't exist in the Figma system, it requires a designer review and either approval as a documented exception or addition to the design system. It should not ship without that review.
Focus rings are on-brand and visible — not suppressed or browser-default.High Tab through every interactive element. Focus rings should use the design's defined focus style, appear on keyboard navigation (and optionally on click, per the spec), and never be outline: none without a custom replacement.
🖼 Images & Media 5 items
Images are sharp on retina/HiDPI displays.High On a retina screen, verify that raster images (JPG/PNG) are served at 2× resolution or delivered via srcset. A 400px image displayed at 200px CSS size will appear blurry on a 2× display. SVGs are inherently resolution-independent and exempt.
Image crop, focal point, and aspect ratio match design intent at all breakpoints. Verify that object-fit and object-position are set correctly so the subject of the image isn't cropped out at narrower viewports. An image that looks correct on desktop can cut off a face or product on mobile.
Broken image state is graceful — the layout doesn't collapse. Remove or block an image request in dev tools and verify the fallback renders correctly: correct dimensions preserved, no layout collapse, alt text visible if applicable. This is especially important for user-generated or CMS-driven image slots.
Decorative SVG icons use aria-hidden="true"; functional SVG icons have labels. SVG icons used purely for decoration should be hidden from screen readers. SVG icons that communicate meaning (status icons, action triggers) need an aria-label or a visually-hidden label.
Images aren't oversized for their container. A 3000px source image served into a 400px container is a performance and quality issue. Verify that image dimensions are reasonably close to their rendered size. Oversized images also sometimes cause subtle layout shifts as they load.
✨ Interaction & Motion 5 items
Transition timing and easing match the motion spec — not defaulted to linear.High Verify that duration and easing curves match the Figma prototype or motion spec. linear easing feels mechanical; most design transitions use ease-in-out or a custom cubic-bezier. Check the computed CSS in dev tools — don't assume it's correct because it animates.
Skeleton loaders and loading states match the design spec in shape and timing.High Throttle the network and verify: skeleton dimensions match the content they're standing in for, shimmer direction and speed match the spec, and the transition from skeleton to content is smooth — not an abrupt swap.
Empty states are designed, not improvised. Every list, feed, table, or data widget has an explicit empty state. Verify illustration, copy, and CTA (if applicable) match the Figma spec. An empty state that reads "No data available" in a system font is a developer placeholder, not a shipped feature.
Error and validation states appear at the correct moment, location, and with approved copy. Inline errors appear below the relevant input; toast/snackbar messages appear at the designed screen position; validation triggers on the correct event (blur vs. submit). Verify all error copy matches approved content — not developer-written fallbacks.
All animations respect prefers-reduced-motion. Enable Reduce Motion in your OS settings and verify that all transitions and animations either stop or are replaced with a simpler, non-motion alternative. This is both an accessibility requirement and a progressive enhancement quality bar.
🧪 Content Resilience 6 items
No lorem ipsum or placeholder content is visible in the build.Critical Any placeholder text, "TBD" copy, stock image watermarks, or dummy data in the build is a pre-flight failure. If this is discovered mid-review, stop and return the build — content resilience testing against fake content produces false confidence.
Minimum and maximum content lengths are tested for every dynamic text element.High Test each text-bearing component at both ends: a very short value (e.g., "Li") and a realistic maximum (e.g., a 40-character name, a 3-line description). The layout should absorb both without breaking, collapsing, or overlapping adjacent elements.
Null and missing field states are graceful.High Remove optional data fields — a missing avatar, a product with no description, a user with no last name — and verify the layout degrades cleanly. Missing optional content should render a neutral fallback, not an empty container, broken layout, or "undefined."
Numbers, currency, and dates are formatted correctly. Verify that large numbers use appropriate separators (1,000 vs 1000), currency symbols and decimal places are consistent, and dates render in the expected format for the target locale. These are often handled in code as raw API output and rarely specced in Figma.
Pluralization and dynamic copy is correct. Any string that changes based on a count — "1 result" vs. "2 results," "1 item in your cart" vs. "3 items" — is grammatically correct in both the singular and plural forms. Hardcoded plural strings are a common, embarrassing oversight.
All approved copy is final — no developer-written fallback strings visible. Scan the built feature for any copy that wasn't in the Figma spec or content doc: button labels, error messages, tooltip text, confirmation dialogs. Developer-authored copy frequently ships without review and erodes brand voice.
📱 Responsiveness 4 items
Step 1 — Verify all breakpoints in browser emulation first.Critical Use Chrome DevTools responsive mode to step through every breakpoint defined in Figma. Confirm layout reflows (column collapses, element hiding, padding adjustments) match the design exactly — same stacking order, same spacing. Also test at 320px, the practical minimum width for modern mobile.
No horizontal overflow at any breakpoint.Critical A horizontal scrollbar at any defined breakpoint is an immediate flag. In emulation, toggle "Show scrollbars" and scan each breakpoint. Horizontal overflow is often caused by a single element — fixed-width container, pre-wrapped text, or non-responsive image — that's easy to miss visually.
Step 2 — Confirm on physical devices after emulation passes.High Emulation doesn't replicate font rendering differences, scroll inertia, tap delay, safe area insets (iPhone notch/home bar), or some CSS rendering quirks. Once emulation passes, verify on at least one iOS and one Android device targeting the most common screen sizes for Golf Pride's audience.
Touch tap targets are minimum 44×44px on mobile — visually smaller is fine, physically larger is required.High All buttons, links, and icon controls must have a physically tappable area of at least 44×44px. The visual element can be smaller; use padding to extend the hit area. Use Chrome's "Emulate touch" and inspect computed sizes.
🌐 Cross-browser Baseline 4 items
Safari (macOS and iOS) is tested explicitly — not assumed to match Chrome.High Safari has a distinct rendering engine (WebKit) and swallows specific bugs: position: sticky failures, flexbox gap issues, gap in older versions, scroll behavior, and date/time input rendering. If the Golf Pride audience includes iOS users, Safari is a primary target, not an afterthought.
Windows font rendering is verified if the audience includes Windows users.Medium ClearType (Windows) renders fonts at lower subpixel quality than macOS. A typeface that looks refined on Mac can appear noticeably heavier or blurrier on Windows Chrome. Test heading typography on Windows — either a physical machine or a Windows VM.
Firefox is spot-checked for layout and form element differences. Firefox renders some form controls (selects, checkboxes, radio buttons, date pickers) with its own default styling that can override custom styles. Verify custom form elements render correctly and that any -webkit--prefixed CSS has Firefox equivalents.
No console errors in any tested browser. Open the browser console and verify that the reviewed feature produces zero errors in Chrome, Safari, and Firefox. JavaScript errors that don't visibly break the UI often indicate broken state management or missing fallbacks that will surface under real user conditions.
⚡ Performance (Visible) 4 items
No visible layout shift on page load.High Load the page on a throttled connection (Chrome DevTools → Slow 3G) and observe whether content jumps as fonts load, images render, or async components insert. Layout shift is a design quality problem — it makes the product feel unstable — as well as a Core Web Vitals metric (CLS).
Web font loading doesn't produce a jarring content flash. Verify that font loading strategy is set to font-display: swap or optional and that the fallback font stack is close enough in metrics that reflow on font load is minimal. A jarring FOUT — where a page snaps from one typeface to another — is a visual quality failure, not just a performance one.
Skeleton/loading states are visible for a perceivably appropriate duration. On a fast connection, skeletons should not flash in and out in under 100ms — that flicker is more jarring than showing nothing. If loading is near-instant, consider whether the skeleton should be suppressed below a minimum threshold, which should be defined in the spec.
Images are appropriately sized for their containers. Check the Network tab in dev tools and verify that image file sizes are reasonable relative to their display dimensions. A hero image over 300–400kb on a production site, or a 2MB image served into a 120px avatar slot, is a visible quality problem — images load progressively and the degraded state is visible to users.
♿ Accessibility 6 items
All interactive elements reachable and operable by keyboard alone.Critical Tab through the entire feature. Every button, link, form control, modal trigger, and menu must be focusable and activatable without a mouse. Tab order must follow the visual reading order — not the DOM order if those differ.
Modal and drawer focus is trapped while open, and returns on close.Critical When a modal opens, keyboard focus must move into it and be contained — tabbing should not reach background content. When the modal closes (via button, Escape, or backdrop click), focus must return to the element that triggered it. This is the most common keyboard accessibility failure in component-rich UIs.
Icon-only buttons and controls have accessible labels.Critical Any button or control with only an icon must have an aria-label or visually-hidden text. Activate a screen reader (VoiceOver on Mac, NVDA on Windows) and navigate to these elements — the label read should be the action, not "button" or silence.
Form inputs have associated labels — placeholder text is not a substitute.Critical Every form field must have a <label> element linked via for/id or aria-label. Placeholder text disappears when the user types, provides no persistent label, and isn't reliably read by screen readers as a field label.
Focus moves to a logical location after async actions.High After form submission, page navigation, or a significant async state change (e.g., a panel loading content), verify that focus moves to a relevant element — the success message, the new content heading, or the next logical step — rather than being lost or staying on a now-irrelevant button.
Run axe DevTools and resolve all Critical and Serious violations before close. Use the axe DevTools browser extension. Resolve every "Critical" and "Serious" violation before design QA closes — these map directly to WCAG 2.1 AA failures. "Moderate" and "Minor" violations are logged in the backlog. Note: axe also catches contrast issues flagged in the Color section — do not double-count.
📝 Forms Conditional 5 items
Browser autofill styling doesn't override the design's input colors.High Browsers inject a yellow or blue background on autofilled fields. Verify that the build overrides :-webkit-autofill styles so autofilled inputs visually match the non-autofilled design. This is a nearly universal oversight that's jarring on first use.
Input types are correctly specified for mobile keyboard optimization. Verify that email fields use type="email", phone fields use type="tel", and numeric fields use type="number" or inputmode="numeric". The correct type triggers the right mobile keyboard — a text keyboard for an email field is a real UX failure.
Disabled and read-only states are visually distinct from each other. Disabled inputs (user cannot interact) and read-only inputs (data is shown but not editable) are frequently confused in implementation. Verify that each has its own visual treatment matching the Figma spec — they should not look identical.
Required field indicators are visible and consistent. Required fields should be marked consistently — asterisk, "Required" label, or per the design spec — on every required field in the form, not just the first one. The pattern should match what's in Figma.
Submission success and failure states are designed and implemented — not improvised.High Verify that both the success state (confirmation message, page redirect, or inline confirmation) and the failure state (server error, network timeout) were specced in Figma and match the build. The submission feedback moment is high-stakes UX and should never be a developer-written fallback.

* The Forms section (dashed border) is conditional — apply it only when the feature being reviewed includes form inputs.
